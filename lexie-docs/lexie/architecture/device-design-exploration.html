01 — Where it goes online

Same URL at home and on the go

v1 is WiFi to the public Lexie service — not Bluetooth for transcripts or TTS. Parents configure the device once; routing changes when the access point does.

🏠 At home

Bookmark joins the household WiFi. Packets route over normal broadband to the deployed Lexie app on your chosen host (for example, Fly, Railway, or a VPS).

https://<your-lexie-host>/v1/… (TLS required)

📱 Away / car / cafe

Parent turns on a phone hotspot. The device reconnects; firmware still points at the same BASE_URL as at home. No BLE data bridge.

One profile · one org child · single age policy
Server: FastAPI · CORS/HTTPS as in SPEC · Device job: POST /explain (audio in → explanation + audio out) · Ops: optional local lexie-ops polls GET /health only
02 — Form factor

Physical design

A slim, book-width clip-on that sits against the page edge. Premium bookmark — not a bright “toy remote” aesthetic. Rely on the teal-green LED ring and matte enclosure.

Front view
Matte soft-touch face · LED ring around the main button · mic under grille (conceptual)
Back view
Back shell · internal antenna keep-out · no kid-facing “spec sticker forest” in production if avoidable
Side profile
~9mm ESP32-S3 class stack · bottom USB-C for charge

Spring clip

Leaf spring at the top slides over book cover or page. Tension tuned for ~3–12mm covers; recess in the body so the profile stays slim.

LED ring (primary output)

Micro-LEDs around the button. Listening = blue pulse. Thinking = teal motion. Speaking = solid teal-green while audio plays. Idle = off. Error = red flash pattern. No always-on “decorative” light.

08 — Ops / cost

Monitoring without another kid-facing app (v1)

A parent or builder can run a small lexie-ops dashboard on a laptop that proxies /health to the public host. Usage and cost truth stay in OpenAI billing + server logs, not on-device counters.

What the ops app shows

Reachability, last deploy, maybe queue depth. Not a second voice surface for the child. Keep it unglamorous: engineering UI is fine for now.

What we do not claim in hardware copy

“100% on-device” or “fully private to the home” is false when STT/LLM/TTS are in the cloud. Be explicit: privacy posture is the usual hosted API story plus your retention settings.
